# Architecture review — pass 1

**Reviewer**: architect-review
**Date**: 2026-05-15
**Verdict**: ACCEPT_WITH_CHANGES

## Summary

The architecture is coherent, well-scoped to the $0/month free-tier constraint, and translates cleanly into a Bicep module map. However it carries one **critical** cross-document inconsistency: the prose in §3 of `03_architecture.md` describes the AI assistant as a single AAD-mapped role (`tcp_ai_assistant`) with no per-user filtering, while the `ai_sequence.mmd` diagram, the `02_database_design.md` RLS design (§9), and the `01_business_requirements.md` security NFRs (NFR-SEC-02) require the Function App to set `SESSION_CONTEXT('aad_object_id')` per-user and rely on RLS for row filtering. Until §3 reconciles RLS, the implementation in Etapa 4–5 is ambiguous and AC-AI-05 is unenforceable. A handful of major issues — DirectQuery vs Import inconsistency, OIDC subject patterns over-broad, missing per-user rate limiting wiring, BACPAC automation under-specified — must also land before IaC starts. With the listed fixes, the design is ready for Etapa 4.

## Major (must fix before IaC starts)

- [ ] **MJ-01** | §6.1 OIDC subject patterns | `gh-feat` uses subject pattern `repo:<owner>/...:ref:refs/heads/feat/*`. GitHub OIDC subject claims **do not support wildcards** in the `ref:refs/heads/*` form; the subject must match exactly, or use the `environment:` claim form. The literal `feat/*` will never match a real workflow token and CI on feat branches will fail OIDC exchange. | All `feat/azure-rewrite` branch deployments will fail with `AADSTS70021: No matching federated identity record found`. | Replace `gh-feat` with either: (a) an `environment:` form federated credential — e.g. `repo:<owner>/<repo>:environment:dev` — and add `environment: dev` to the workflow; or (b) a pull_request claim if all feat-branch CI flows through a PR. Document that wildcards in `ref` are not honored by Entra federated credentials.

- [ ] **MJ-02** | §5 RBAC matrix — GitHub Actions SP scope | The OIDC SP holds `Contributor` at RG scope **plus** `Key Vault Secrets Officer` **plus** `SQL Server Contributor` **plus** `Reader` at subscription. `Contributor` at RG already includes everything the two RG-scoped specialist roles grant; the specialist rows are redundant and bloat the principle of least privilege. Also, `SQL Server Contributor` lets CI delete the SQL server — a high-blast-radius capability used only for a transient firewall rule. | Over-granted CI tokens are a top-5 cloud breach vector. A leaked OIDC token (e.g., via a malicious dependency that prints `$ACTIONS_ID_TOKEN_REQUEST_TOKEN`) gives the holder full RG control. For a single-user thesis project the risk is small, but the precedent is bad and the rubric for the security reviewer will flag it. | Drop `Key Vault Secrets Officer` and `SQL Server Contributor` (subsumed by `Contributor`). Replace `Contributor` with a narrower composite: `Website Contributor` + `SQL DB Contributor` + `Storage Account Contributor` + `Monitoring Contributor` + a custom `KeyVaultDeployer` role that allows `Microsoft.KeyVault/vaults/write` and `…/secrets/setSecret/action` but not `…/secrets/delete` or `…/purge/action`. Keep subscription `Reader`. Document the trade-off in an ADR.

- [ ] **MJ-03** | §3.2 step 4 / §8.3 row "SQL injection through generated queries" | The text says the LLM returns "a SQL query (or refusal)" and `safe_query.py` enforces "parameterized" queries. This is internally contradictory: an LLM-emitted SQL string is by definition string-built, not parameterized — `safe_query.py` cannot retroactively parameterize an arbitrary SELECT. The real contract must be: the LLM emits a **structured query intent** (e.g., `{view: "v_employee_performance", filters: [...], group_by: [...], order_by: [...], limit: 1000}`) which the function maps to a parameterized SQL via a small server-side builder. | If the LLM is allowed to emit raw SQL, the only validation possible is AST-walking with allowlists — workable, but fragile (e.g., an `OR 1=1` snuck inside a `CASE WHEN` branch). Worse, the doc claims both "AST-validated" and "parameterized" without explaining how arbitrary SQL becomes parameterized. The thesis security reviewer will pick this apart. | Either: (a) commit to a **structured-intent contract** (recommended) and document the JSON schema the LLM must emit in §3.2 and in `safe_query.py`'s contract; or (b) keep raw SQL but be explicit: "the LLM's output is treated as untrusted SQL; `safe_query.py` parses it with `sqlglot`, allowlists tables `v_*`/`dim_*`, allowlists statement types `SELECT` only, forbids `UNION`/`INTO`/`WAITFOR`/`xp_*`/`OPENROWSET`, enforces `TOP <= 1000`, and rejects anything that does not parse cleanly". Add a CI test fixture of 20+ adversarial prompts in Etapa 5.

- [ ] **MJ-04** | §3.2 step 3.6 + §10 cost model + §8.3 threat row | "Results … are returned to the browser … The JS renders a table + a one-paragraph natural-language summary produced by a **second Anthropic call**". A second Anthropic call per `/api/ask` doubles the per-request token cost and the per-request latency, and is not reflected in the §10 expected-use math (which assumes ~50 questions/day, presumably one Anthropic call each). The DoS mitigation in §8.3 ("10 questions/min/user; circuit-breaker at 1 000 tokens/question") also does not capture the second call. | Cost projections are off by ~2x for Anthropic API spend (the bottleneck for "cost discipline"). The single-call design is strongly recommended.

- [ ] **MJ-06** | §15 R1 + §14 cold-start budget | The mitigation for SQL auto-pause cold-start is "ping `/api/ask` from a synthetic monitor every 15 min during 09:00–18:00 RO". - [ ] **MJ-07** | §8.2 + §3.4 — `x-ms-client-principal` forgeability | "the raw Function URL is omitted from documentation and clients.
