# Etapa 4 security review — pass 1

**Reviewer**: security-auditor
**Date**: 2026-05-16
**Verdict**: ACCEPT_WITH_CHANGES

## Summary

The Etapa-4 IaC and Function App scaffold gets the high-stakes posture right: zero plaintext secrets in source or Bicep outputs, system-assigned MI everywhere, OIDC-only CI/CD, `httpsOnly`/`minTlsVersion 1.2`/`ftpsState Disabled` on the Function App, KV-RBAC with diagnostic settings, and constant-time secret comparison on `/api/ask`. There are however four production-blocking defects that will break the smoke test or leave a sensitive failure window open: an output-name mismatch that empties the smoke `curl` URL, an unsafe RLS-disable window with no transactional roll-back guarantee, a missing `staticwebapp.config.json` (so SWA never injects the shared-secret header), and an environment-variable name mismatch between `bacpac_export.py` and `functions.bicep`. Several major hygiene issues round out the list (unquoted secret expansion on the runner, `bandit` not scanning `function_app/`, missing GitHub `environment:` blocks referenced by federation subjects, `iac-whatif` without `--no-pretty-print` quoting safety, etc.). After CR-01 through CR-04 are addressed, the posture is acceptable for the thesis-context production deploy.

## Critical (blocks production deploy)

- [ ] **CR-01** | `.github/workflows/cd.yml:65` | The smoke job extracts `AZURE_FUNCTION_APP_HOSTNAME` from `azd env get-values`, but `infra/main.bicep:265` emits `AZURE_FUNCTION_APP_DEFAULT_HOSTNAME`. The grep returns empty → `function_app_hostname` step output is `""` → `curl -fsS "https:///api/ping"` resolves to an empty host and either fails to dial or hits an arbitrary local resolver. | Threat: a green CI run despite a broken/missing Function App (false-positive deploy success); in the worst case the request lands on whatever name the runner resolves for an empty host, leaking the OIDC-authenticated session's `Host` header to a wrong endpoint. | Fix: change line 65 to `grep AZURE_FUNCTION_APP_DEFAULT_HOSTNAME` (or rename the Bicep output). Also assert non-empty before curling (`[[ -n "$HOSTNAME" ]] || exit 1`).

- [ ] **CR-02** | `infra/scripts/postprovision.ps1:60-89` and `infra/scripts/postprovision.sh:70-95` | RLS policy `rls.TradesAccessPolicy` is turned OFF, an `INSERT` runs, then the policy is turned back ON — all in a single `sqlcmd` batch with `-b`. There is no T-SQL `BEGIN TRY ... BEGIN CATCH ... ALTER SECURITY POLICY ... STATE = ON ... END CATCH; THROW;` guard, no PowerShell `try { ... } finally { ... }` re-enable, no `trap` in sh. If `sqlcmd` is interrupted (SIGINT, runner timeout, network blip after `STATE = OFF`), or the `INSERT` fails on `-b` after the first `ALTER`, the policy stays OFF. Worse, the PS catch block at line 90 only warns ("Continuing..."); the script then proceeds to flip the server to AAD-only, leaving a policy-disabled database admissible to every MI/admin token holder until manual intervention. | Threat: a transient runner failure between `STATE = OFF` and `STATE = ON` produces a long-lived data-plane RLS bypass on `fact_Trades`. Per ADR-003 §4, deny-by-default is the design intent; this implementation violates it. | Fix: wrap the T-SQL in `BEGIN TRY ... END TRY BEGIN CATCH ALTER SECURITY POLICY rls.TradesAccessPolicy WITH (STATE = ON); THROW; END CATCH;`. Additionally, add a final defensive `ALTER SECURITY POLICY rls.TradesAccessPolicy WITH (STATE = ON);` as a separate `sqlcmd` invocation in a PowerShell `finally` / sh `trap '... ' EXIT` block. Treat the `INSERT` failure as fatal — do not swallow with `warn` and continue to the AAD-only flip.

- [ ] **CR-03** | (missing file) `static/staticwebapp.config.json` | The Function App `/api/ask` validator (`function_app/triggers/ask.py:82-103`) refuses any request that does not present a matching `X-SWA-Forwarded` header (403). Document the wiring in `03_architecture.md §8.2 bullet 4`. If this is intentionally deferred to Etapa 5, gate the `/api/ask` 403 path behind a feature flag so the smoke test does not regress.

- [ ] **CR-04** | `function_app/triggers/bacpac_export.py:50` vs `infra/modules/functions.bicep:177` | The trigger reads the SQL admin password from env var `SQL_ADMIN_PASSWORD_EXPORT`, but Bicep populates `TCP_SQL_ADMIN_PASSWORD` with the `SQL-ADMIN-PASSWORD-EXPORT` KV reference. ## Major

- [ ] **MJ-01** | `.github/workflows/cd.yml:55,105` | `azd env set ANTHROPIC_API_KEY ${{ secrets.ANTHROPIC_API_KEY }}` is unquoted. If the secret value ever contains whitespace, a `$`, a backtick, or a `;` the shell will split / interpret it before `azd` sees it — corrupting the value and at worst leaking fragments into the runner's process table. Anthropic keys are currently `sk-ant-…` with no whitespace, but the contract should be defensive. | Threat: silent secret corruption → 401 from Anthropic at runtime; metadata leakage via `ps auxf` if a value contains shell metacharacters. | Fix: quote — `azd env set ANTHROPIC_API_KEY "${{ secrets.ANTHROPIC_API_KEY }}"`. Better: pass via env var, `env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}` then `azd env set ANTHROPIC_API_KEY "$ANTHROPIC_API_KEY"`, so GitHub's secret masker can suppress it from logs.

- [ ] **MJ-02** | `.github/workflows/ci.yml:144` | `bandit -r tcp -lll -iii` scans only `tcp/`. The Function App code in `function_app/triggers/*` (which now contains HMAC validation, header parsing, and a future LLM call site) is **not** scanned. Spec at the top of the audit checklist (#8) explicitly requires `bandit` to cover "Function App + tcp/* sources". | Threat: a future Etapa-5 commit that introduces e.g. `subprocess.Popen(shell=True)` in a trigger ships without detection. | Fix: `uv run bandit -r tcp function_app -lll -iii`.

- [ ] **MJ-03** | `.github/workflows/ci.yml` (whole file) and `cd.yml` (whole file) | Neither workflow declares `environment:` on any job, yet `03_architecture.md §6.1` configures the OIDC federation credential `gh-dev` against `subject: repo:<owner>/...:environment:dev`. As written, no CI job will ever match `gh-dev`; pushes to `feat/**` (line 7 of `ci.yml`) match neither `gh-main` (refs/heads/main) nor `gh-pr` (pull_request). The `iac-whatif` job at line 190 only runs on `pull_request`, so it survives; everything else relies on no Azure call from `feat/**`. This works today but the §6.1 contract drifted from reality — and on the next branch that tries `azd what-if`, OIDC will silently 401. | Threat: configuration drift between Entra ID federation credentials and workflow subject claims; future "why doesn't OIDC work?" debugging session. | Fix: either drop `gh-dev` from §6.1 if no `environment:` flow is intended for Etapa 4, or add `environment: dev` to the relevant job in `ci.yml` and create a corresponding GitHub `Environment` with the same name. Document in §6 which workflow trigger maps to which federated credential.

- [ ] **MJ-04** | `infra/modules/sql.bicep:99-109` | The `AllowAllAzureServices` firewall rule (`0.0.0.0 → 0.0.0.0` virtual range) is documented in `03_architecture.md §8.2` but the threat-model entry is incomplete. This rule permits **every** Azure tenant's outbound traffic, not just our Function App MI — any compromised Azure VM anywhere can attempt SQL-server TLS handshakes (auth still required, but the surface is the whole Azure backbone). Free-tier constraint is real, but the comment at line 99 ("Required by the free-tier design") under-sells the trust assumption. | Threat: cross-tenant SQL probing; brute-force against `tcpadmin` while SQL-auth is still enabled (i.e., between schema apply and the AAD-only flip in postprovision). | Fix: (a) annotate the resource with the full threat assertion ("trusts AAD-only flip + strong password during the bootstrap window"); (b) verify the postprovision flip completes within minutes of the deploy and document the bootstrap-window duration in `docs/security/bootstrap_window.md`; (c) consider adding a TODO to swap to a service-tag-restricted firewall rule once Functions support outbound static IPs (Flex Consumption).

- [ ] **MJ-05** | `infra/modules/storage.bicep:48` | `allowSharedKeyAccess: true` is required by the Functions runtime (`AzureWebJobsStorage` uses the connection string), but it also enables any holder of the storage key (KV `STORAGE-CONNECTION-STRING` secret) to read every blob in the account — including the future `bacpac-exports` container. The MI's BACPAC role is scoped narrowly (line 167-177), but the key path bypasses that scoping entirely. | Threat: a leaked storage-account key (or a KV compromise) grants full account read/write, exfiltrating every BACPAC. | Fix: post-Etapa-5, migrate `AzureWebJobsStorage` to MI-based identity (`AzureWebJobsStorage__credential=managedidentity`) and set `allowSharedKeyAccess: false`. Until then, document the trade-off in `03_architecture.md §8.3` as an explicit threat row.

- [ ] **MJ-06** | `infra/modules/keyvault.bicep:66,84` | `enablePurgeProtection: false` is a thesis-context trade-off so `azd down` can fully tear the vault. The flag is documented; what is **not** documented is the resulting blast-radius: an authorized `Key Vault Secrets Officer` (the OIDC SP per line 211-219) can `az keyvault secret delete` + `az keyvault secret purge` to obliterate every secret with no recovery window. Combined with the 7-day soft-delete retention, an accidental `purge` is unrecoverable. | Threat: a compromised OIDC token, or a buggy CI step running `az keyvault secret purge`, irreversibly destroys the production secrets in two API calls. | Fix: keep `enablePurgeProtection: false` for the thesis cycle, but explicitly remove `Key Vault Secrets Officer` from the OIDC SP after Etapa 4 is provisioned (set `oidcPrincipalId: ''` on subsequent `azd provision` runs). Document the "flip to true post-defense" reminder in `STATE.md`.

- [ ] **MJ-07** | `infra/modules/keyvault.bicep:88-94` | `publicNetworkAccess: 'Enabled'` and `networkAcls.defaultAction: 'Allow'` together mean the vault is reachable from **any** IP on the public internet (AAD auth still required, but the TLS hand-off plus token endpoint are public). The §8.1 free-tier rationale stands, but the dual `Allow` is broader than needed: `bypass: 'AzureServices'` is honoured only when `defaultAction = 'Deny'`. Setting `defaultAction: 'Allow'` makes the `bypass` line a no-op and explicitly opens the vault to non-Azure IPs. | Threat: token-replay or AAD-token-theft attacks now have the global internet as their entrypoint; if the OIDC SP token leaks (e.g., from a runner artifact), an attacker can read every secret without first compromising an Azure footprint. | Fix: switch to `defaultAction: 'Deny'` + `bypass: 'AzureServices'`. Add the developer's home IP to `networkAcls.ipRules` for break-glass (or use `az keyvault network-rule add` outside Bicep) during the thesis cycle; document in `docs/security/break_glass.md`.

- [ ] **MJ-08** | `infra/modules/functions.bicep:165-167` | The empty `TCP_GENERATOR_OID` app setting (`value: ''`) is populated later by the postprovision script via `az functionapp config appsettings set`. Between the Function App startup and the postprovision completion, the daily-generator's `tcp.synth.run_daily()` (if it fires) will read an empty OID, call `sp_set_session_context @key=N'aad_object_id', @value=NULL`, and per ADR-003 §3 the RLS predicate's deny-by-default kicks in — the generator will write zero rows silently. The `run_on_startup=False` on every timer mitigates this on a normal deploy but not on a manually-triggered run before postprovision finishes. | Threat: race condition between `azd provision` and `postprovision`, masked by silent RLS denial; operator sees the function "succeed" but no rows are inserted. | Fix: add a startup check in `tcp/db.py` (or in each timer entrypoint) that refuses to proceed if `TCP_GENERATOR_OID` is empty / not a valid GUID, raising a clear `RuntimeError('postprovision incomplete')`. ## Threat surface delta vs Etapa 3

Etapa 3 ran the synthetic-trade generator on a local SQL Server with no public endpoints. The cloud deploy introduces six new threat vectors, each documented and either mitigated or accepted:

1. **Public SQL endpoint with `AllowAllAzureServices` firewall** — every Azure tenant's outbound IP can dial the server's TLS endpoint. Mitigation: AAD-only auth after bootstrap (postprovision step 3); diagnostic SQL audit events ON; bootstrap-window minimised. **Residual risk**: a brute-force attempt against `tcpadmin` succeeds during the bootstrap window if the password is weak — mitigated by the GUID-shaped default (~120 bits entropy) and the postprovision flip running within minutes.

2. **Public Key Vault endpoint with `defaultAction: 'Allow'`** — global-internet AAD token endpoint. Mitigation: RBAC-only, no access policies; OIDC SP role is the narrower `Secrets Officer` (not `Administrator`); diagnostic `AuditEvent` ON. **Residual risk** (MJ-07): OIDC token theft → full secret read; recommend `Deny` with `bypass: 'AzureServices'`.

3. **Function App `/api/ask` direct URL** — the raw `*.azurewebsites.net/api/ask` endpoint is reachable from anywhere. Mitigation: `x-ms-client-principal` 401 + `X-SWA-Forwarded` 403 (constant-time compare). **Residual risk** (CR-03): the SWA-side injection of `X-SWA-Forwarded` is not yet implemented; until it is, the 403 path is the only outcome for any legitimate caller, and direct callers can probe but not authenticate.

4. **CI/CD runner has plaintext secret in process memory** — `azd env set ANTHROPIC_API_KEY` writes the value to `~/.azd/environments/.../config` on the runner. Acceptable for short-lived GitHub-hosted runners (`ubuntu-24.04` ephemeral) but a self-hosted runner would inherit the risk indefinitely. Mitigation: ephemeral runners only; GitHub secret masker redacts the value from logs. **Residual risk** (MJ-01): unquoted expansion can leak fragments into the runner's process table.

5. **Storage account holds BACPAC backups** — every weekly export is a full DB snapshot accessible via the storage key. Mitigation: container-scoped MI permissions; lifecycle rule deletes blobs >28 days old; `allowBlobPublicAccess: false`. **Residual risk** (MJ-05): the storage key in KV bypasses container scoping; migrate to MI-based `AzureWebJobsStorage` post-Etapa 5.

6. **Postprovision script is a privileged one-shot under OIDC SP context** — it runs `Set-AzSqlServerActiveDirectoryOnlyAuthentication`, deletes KV secrets, and writes the MI OID into `dim_UserRoles`. A bug here can leave the database in an inconsistent state (CR-02). Mitigation: idempotent design, explicit verification steps 5 & "Verify bootstrap secret" at the tail. **Residual risk** (CR-02): no atomic guarantee that the RLS policy is re-enabled on script crash; fix requires a T-SQL `TRY/CATCH` plus a PowerShell `finally`.
