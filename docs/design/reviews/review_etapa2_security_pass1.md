# Etapa 2 security review — pass 1

**Reviewer**: security-auditor
**Date**: 2026-05-15
**Verdict**: ACCEPT_WITH_CHANGES

## Summary

The Etapa 2 schema, Python connection layer, and SQL tests implement the ADR-003
RLS contract correctly, with a strong deny-by-default posture, fully
parameterised dynamic SQL, and a try/finally lifecycle that resets
`SESSION_CONTEXT` even on exceptions. The threat surface is small (synthetic
data, single MI, AAD-only auth post-bootstrap) and the role split between
`tcp_ai_assistant` and `tcp_generator` is correctly narrow. Three issues
require fixing before merge: the gitleaks allowlist does not cover
`db/README.md` (a real path that contains the same dev-only literal),
several CI security gates are configured to warn-only (`bandit`, `pip-audit`,
`sqlfluff`) when ADR-spec calls them out as fail-gates, and `gitleaks-action`
is pinned to a major tag rather than a full SHA. None of the issues affect
production data confidentiality, but two of them weaken the documented CI
posture in `03_architecture.md §9.1`.

## Critical (blocks merging Etapa 2)

- [ ] **CR-01** | `.gitleaks.toml:8-11` | The allowlist `files` array lists only `docker-compose.dev.yml` and `docs/dev_setup.md`, but the literal `YourStrong!Passw0rd` also appears at `db/README.md:32,35,38` (three occurrences). | Threat model: a `gitleaks` run on the full repo will fail in CI, blocking otherwise-clean PRs, **or** if the gitleaks rule for "generic-api-key" / strong-password matches, contributors will be tempted to widen the allowlist with a global regex — exactly the anti-pattern this configuration was designed to avoid. Even worse, a future contributor may move the placeholder to a real config file and the existing allowlist regex (`YourStrong!Passw0rd` is a pure literal under `regexes`, applied repo-wide) will silently mask it. | Fix: add `"db/README.md"` to the `[allowlist].files` array. Convert the `regexes` entry to a stopwords array (`stopwords = ["YourStrong!Passw0rd"]`) **scoped via `paths`** so the literal is only allowlisted on the three specific files, not globally. Add a CI test (`scripts/verify_gitleaks_allowlist.py`) that asserts the placeholder appears only in the three allowlisted files.

- [ ] **CR-02** | `.github/workflows/ci.yml:92,111,114` | Three security gates are configured to **never fail the build**: `sqlfluff lint ... 2>&1 || true`, `bandit -r tcp -ll || true`, `pip-audit --strict || true`. `03_architecture.md §9.1` (the canonical CI contract) declares these as `fail` gates ("any rule violation → fail", "any High severity → fail", "any CVE with severity ≥ High → fail"), as do `17.1` lines 728-729 in the Etapa-4 acceptance checklist. | Threat model: a future High-severity bandit finding (e.g., a `subprocess.run(shell=True)` slipped into `tcp/`) or a freshly-disclosed High CVE in `pyodbc`/`azure-identity` will not block merging, and the policy gap is silent (CI shows green). This is the textbook "security misconfiguration" (OWASP A05:2021). | Fix: remove the `|| true` from the bandit and pip-audit steps. For sqlfluff, either remove `|| true` once the lint baseline is clean or scope the suppression to specific rules with a `.sqlfluff` config file. Document the change as part of the §9.1 contract.

## Major

- [ ] **MJ-01** | `.github/workflows/ci.yml:106` | `gitleaks/gitleaks-action@v2` is pinned to a major-version tag (mutable), not a full commit SHA. | Threat model: a compromised third-party action could exfiltrate `${{ secrets.GITHUB_TOKEN }}` (which has `contents: read` + `id-token: write` here) or smuggle code into the build artifact. GitHub's security guidance for third-party actions is "pin to a full-length SHA" precisely because tag mutability is a supply-chain attack vector (e.g., the `tj-actions/changed-files` incident, 2025-03). The first-party `actions/*` references at lines 23, 26, 31, 52, 56, 60, 71, 82, 86 are acceptable on a major tag (the Microsoft maintenance posture is stricter), but a third-party action with access to `id-token: write` deserves the SHA pin. | Fix: replace `gitleaks/gitleaks-action@v2` with `gitleaks/gitleaks-action@<full-sha>` (look up the current SHA for v2 release). Also pin `astral-sh/setup-uv@v3` to a SHA for the same reason. Add a Dependabot config so SHAs are auto-bumped on each release.

- [ ] **MJ-02** | `tests/sql/test_rls_smoke.sql:66,90,128,143,148` | The smoke test calls `sp_set_session_context @key=N'aad_object_id', @value=..., @read_only=0`. ADR-003 §2 mandates `@read_only=1` on the production code path so the value cannot be overwritten mid-batch. The test bypasses this lock so it can flip between scopes within one transaction, but the test does not include a **dedicated TC-4** that exercises the `@read_only=1` lock semantics (i.e., a successful set, followed by a second `sp_set_session_context` call that must fail with error 15664 "Cannot set the value of the read-only key"). | Threat model: a future regression where `tcp/db.py` accidentally emits `@read_only=0` would not be caught by the test suite. The Python unit test `test_connection_for_user_sets_and_resets_context` checks the SQL **string** contains `@read_only = 1`, but no integration test confirms SQL Server actually enforces the lock at runtime. | Fix: add `TC-4 read_only_lock` to `test_rls_smoke.sql` that issues `sp_set_session_context ... @read_only=1` then attempts a second set without `@read_only=0` and expects the call to raise. Update ADR-003 §7 to list this as a fourth CI test case.

- [ ] **MJ-03** | `tcp/db.py:222` | The reset call on exit is run on the **same cursor** that may have executed the user's last statement. If the user's body raises an exception *and* the cursor is in an unrecoverable error state (e.g., a transaction was aborted by `XACT_ABORT`), `cursor.execute(_SQL_RESET_CONTEXT)` will itself fail. The current code logs the failure but does **not** close the connection synchronously — it falls through to `conn.close()`, which Microsoft-ODBC will perform but pyodbc connection pooling could potentially recycle a connection that still has `SESSION_CONTEXT('aad_object_id')` set if `conn.close()` returns the connection to a process-level pool without resetting it. | Threat model: in production the Function App runs `pyodbc` with `pyodbc.pooling = False` by default (correct), and the ODBC driver pool (`Pooling=False` in the connection string is **not** set) **may** keep the connection in the driver-level pool. If a pooled connection retains `SESSION_CONTEXT('aad_object_id')`, the next request inherits the previous user's identity — cross-tenant leakage. ADR-003 §4 documents this risk; the code does not pin the mitigation. | Fix: (a) add `Pooling=False;` to `build_connection_string` for the Function App auth modes (`AAD_MANAGED_IDENTITY`, `AAD_DEFAULT`), accepting the small reconnect cost for safety; **or** (b) keep ODBC pooling but issue `EXEC sp_reset_connection` (T-SQL `sys.sp_reset_connection` is not directly callable; use `pyodbc.Connection.set_attr(SQL_ATTR_RESET_CONNECTION, 1)` on close) so the server resets `SESSION_CONTEXT` for the next checkout. Document the chosen mitigation in ADR-003 §4 and add a CI integration test that opens two consecutive connections with different `oid` values and asserts no leakage on the second.

- [ ] **MJ-04** | `tcp/db.py:167-192` (`open_connection`) | The function is public, has no caller guidance beyond a docstring sentence ("Use this only for infrastructure tasks"), and **bypasses the SESSION_CONTEXT setup**. ADR-003 §3 relies on deny-by-default if a contributor forgets to use `connection_for_user`, but `open_connection` is the exact escape hatch that a future contributor could grab from autocomplete. The deny-by-default safety net is real (the RLS predicate returns 0 rows), but a contributor running an **INSERT** via `open_connection` for what they think is a benign maintenance task could bypass the BLOCK PREDICATE iff their MI happens to be admin-scoped — and the daily generator MI is admin-scoped (ADR-003 §5). | Threat model: defence-in-depth gap. The control fails open for `tcp_generator`-scope operations. | Fix: (a) rename `open_connection` to `_open_raw_connection` and add a module-level `__all__` that does not export it; (b) add an opt-in flag `bypass_session_context: bool = False` whose default is `False` and which raises `AuthError` unless explicitly set to `True`; (c) add a comment block citing ADR-003 §4 directing all user-driven paths to `connection_for_user`; (d) add a `scripts/lint_db_callers.py` step in CI that greps for `open_connection(` outside `tcp/db.py`, `tests/`, and a documented allowlist of admin scripts.

## Minor / nits

- [ ] **MN-01** | `db/migrations/V001__init.sql:1278` | `schema_history.checksum` is stored as the literal `'TODO-checksum-set-by-CI'`. The `db/README.md` states "CI computes the checksum and verifies the recorded value against the script-on-disk to detect drift" — until the CI step lands (Etapa 4), this column is a `TODO` that could mask a drifted production migration. | Fix: track in `docs/decisions/ADR-004-bacpac-export-schedule.md` (or open ADR-005) the explicit deadline (Etapa 4) by which the checksum is no longer a `TODO`. Add `IF @checksum LIKE 'TODO%' RAISERROR(...)` at the head of every forward script from V002 onward.

- [ ] **MN-02** | `db/migrations/V001__init.sql:1207-1213` | The security policy applies block predicates `AFTER INSERT` and `AFTER UPDATE` but **not** on `BEFORE/AFTER DELETE`. Per the Microsoft docs (RLS BLOCK PREDICATE), `DELETE` is also bound by the FILTER predicate, so rows the caller cannot SELECT cannot be deleted either — and the BLOCK PREDICATE for DELETE is "after delete", which is symmetric. The current grant matrix gives no role the right to DELETE on `fact_Trades`, so this is defence-in-depth only. | Threat model: if a future migration adds DELETE rights (e.g., a soft-delete migration for GDPR purposes) without re-reading this section, the BLOCK PREDICATE will not catch cross-trader DELETEs through admin-scope MIs (FILTER predicate still applies, but a future bug in scope resolution could leak). | Fix: extend the policy with `ADD BLOCK PREDICATE rls.fn_TradesPredicate(trader_id) ON dbo.fact_Trades AFTER DELETE` (and `BEFORE DELETE` is implicit via FILTER). Same for `fact_DailyTraderPnL`. Document the choice in `02_database_design.md §9.3`.

- [ ] **MN-03** | `tcp/db.py:58` | `class ConnectionError(TcpDbError):  # noqa: A001 — domain-specific name; shadow is intentional.` shadows the Python builtin `ConnectionError`. The `noqa` is documented but `bandit`/`pylint` will still occasionally surface this. The downstream consequence is a contributor doing `except ConnectionError as exc` may accidentally catch the wrong type. | Fix: rename to `TcpConnectionError` (the integration test already imports it that way at `tests/integration/test_db_smoke.py` — but wait, that test imports `from tcp.db import ConnectionError as TcpConnectionError`, confirming the foot-gun). Make the rename in `tcp/db.py` and drop the alias from the test.

- [ ] **MN-04** | `tests/integration/test_db_smoke.py:59-64` | The cursor.execute uses a parameter (`?`), correctly. But the SQL string concatenation pattern `"INSERT INTO dbo.dim_UserRoles ... VALUES (?, NULL, 'admin', 1)"` interleaves literal values with the parameter. For this fixture (a constant `'admin'`) it is fine; flag as a "model-after-good-practice" reminder so future contributors don't take this line as a template for variable scope values. | Fix: add a comment `# All variable values must be parameter-bound; only the constant scope 'admin' is inlined.` above the call.

- [ ] **MN-05** | `docker-compose.dev.yml:7` | `MSSQL_SA_PASSWORD: "${TCP_SQL_DEV_PASSWORD:-YourStrong!Passw0rd}"` defaults to the well-known literal **if** the env var is unset. A developer who forgets to export the var will spin up a container with the publicly known password. There is no host-port firewall hint in `docs/dev_setup.md` to remind them not to expose port 1433 publicly. | Threat model: a developer running this on a public cloud VM (rare but possible) would expose a SQL Server with a guessable SA password. The port mapping `1433:1433` binds to all interfaces by default on Docker. | Fix: change the binding to `127.0.0.1:1433:1433` to enforce localhost-only. Optionally, remove the default value (`${TCP_SQL_DEV_PASSWORD:?TCP_SQL_DEV_PASSWORD must be set}`) so the compose fails fast when the env var is missing.

- [ ] **MN-06** | `docs/dev_setup.md:54,69,77,109,118-120` | Six occurrences of `YourStrong!Passw0rd` in shell snippets. The gitleaks allowlist covers `docs/dev_setup.md` already (good), but the document does **not** caution the reader to choose a different password before exposing the host externally — only the compose default uses the literal. | Fix: prepend a yellow "Local-only password — DO NOT use in any shared environment" note to the `One-Time Setup` section.

- [ ] **MN-07** | `pyproject.toml:50` | `ignore = ["S608"]` (parameterised SQL false positive) is repo-wide. Once `safe_query.py` lands (Etapa 5) the rule should be re-enabled on `tcp/safe_query.py` to catch any string-formatted SQL that slips past `sqlglot` parsing. | Fix: add `tcp/safe_query.py = []` to `tool.ruff.lint.per-file-ignores` when the file is added in Etapa 5 (informational; not a blocker for Etapa 2).

- [ ] **MN-08** | `.github/workflows/ci.yml:78,98` | `sql-lint` and `secret-scan` jobs do not declare `permissions:`. They inherit the workflow-level block (`contents: read`, `id-token: write`, `pull-requests: read`). `id-token: write` is unnecessary for these two jobs (no OIDC exchange happens). Best practice is to declare the minimum per-job. | Fix: add `permissions: { contents: read }` to the `sql-lint` and `secret-scan` jobs explicitly.

- [ ] **MN-09** | `tcp/db.py:210` | The log emits only `oid_str[-4:]`. ADR-003 §3 confirms forensic chain is intact via SQL audit logs. For local dev debugging, a contributor may temporarily widen the log; document the rule "never log the full `aad_object_id` outside SQL audit" as a comment near the bind call. | Fix: one-line comment above `log = _log.bind(...)`.

- [ ] **MN-10** | `tests/sql/test_rls_smoke.sql:62-66` | The test inserts an `admin`-scope row, sets the session context, inserts two `fact_Trades`, then switches scope to `trader`. There is no assertion that the **block predicate fired** during the `trader`-scope phase (i.e., no negative test that proves a trader cannot INSERT a fact row attributed to a different `trader_id`). The current TC-2 only proves the FILTER predicate. | Fix: append a `TC-2b block_insert` case that, while session context is the principal's trader oid, attempts `INSERT INTO dbo.fact_Trades ... (trader_id = @other_emp_id, ...)` and asserts the INSERT fails with error 33504 ("The attempted operation failed because the target object ... is referenced by a security policy.").

## Threat surface delta vs Etapa 1

Etapa 1 was design-only — no executable code. Etapa 2 adds:

1. **SQL DDL on a real database** — RLS policy + grants + four custom roles. New attack surface: a misconfigured grant or predicate could leak rows. Mitigated by the role-split design (`tcp_ai_assistant` cannot read `fact_Trades` directly) and the deny-by-default predicate. Net delta: **low risk increase**, well covered by `test_rls_smoke.sql`.
2. **Python module `tcp/db.py`** — handles connection-string composition (with passwords on the dev path), session-context lifecycle, and logging. New attack surface: connection-string injection if env vars were attacker-controlled (they are not; CLAUDE.md "credentials handling" policy), pool-leakage if a connection is reused without context reset (MJ-03 addresses this), and incorrect lifecycle on exceptions (covered by `test_connection_for_user_resets_even_when_body_raises`). Net delta: **medium risk increase**, mitigated by tests + lifecycle contract.
3. **Docker compose for local SQL** — exposes port 1433 with a default password. New attack surface: a developer running on a public host. MN-05 reduces the surface to localhost-only. Net delta: **low risk increase**, geographically bounded to dev workstations.
4. **CI workflow `ci.yml`** — gitleaks/bandit/pip-audit gates that should fail-stop. CR-02 documents the warn-only misconfiguration. Net delta: **low risk increase** while the gates remain warn-only; converts to **risk reduction** once CR-02 is fixed.
5. **Dependencies**: `pyodbc`, `azure-identity`, `pydantic`, `polars`, `structlog`. All current as of 2026-05-15. No critical CVEs. `pip-audit` (once made fail-gating) will catch regressions.

No new secrets are introduced in source. Only a single dev-only placeholder password, already correctly allowlisted for two of three locations (CR-01 patches the third).

## Recommendation

**ACCEPT_WITH_CHANGES.** Land CR-01 and CR-02 before tagging `v1.0-db-ready` and merging the Etapa 2 PR to `main`. MJ-01/02/03/04 are tracked-for-next-iteration (open issues against Etapa 4 milestone) — none of them affect Etapa 2 correctness, but MJ-03 in particular must be closed before any production traffic touches the Function App in Etapa 5. The Minor/nits list can be batched into a single follow-up PR with no urgency.

The RLS contract from ADR-003 is implemented faithfully. The role split is well-considered. The Python lifecycle handles exceptions correctly. The defense-in-depth gaps that remain are documented (MJ-03 already called out in ADR-003 §4 as a known risk) and trackable.
